## BLOQUE A — FIX GOOGLE TAG

### Diagnóstico
Grep en `layout/theme.liquid` + repo-wide:
- `G-EEFRE4XFMF` → **NO aparece** en ningún `.liquid` del tema.
- Lo único hardcodeado es el contenedor **GTM-NPCDS6X** (líneas 4-10 de `theme.liquid`), que es el contenedor **zombie** que el prompt prohíbe tocar. Está colocado al principio del `<head>`, ANTES de `<meta charset="utf-8">`.

Verificación en producción (`curl https://www.encopadebalon.com/`):
- `G-EEFRE4XFMF` aparece **8 veces**, todas dentro de un JSON de configuración del **Web Pixel Manager nativo de Shopify** (`wpm`), bajo `"google_tag_ids":["G-EEFRE4XFMF","AW-860467003","GT-NNVDP65S"...]` con `gtag_events` (begin_checkout, view_item, purchase, page_view, add_payment_info...).
- **NO existe** ningún `<script src="...googletagmanager.com/gtag/js?id=G-EEFRE4XFMF">` hardcodeado en el head.

### Conclusión → CAMINO (b)
El tag GA4 `G-EEFRE4XFMF` lo inyecta la **integración nativa Shopify → Google & YouTube** a través del sistema de Customer Events / Web Pixel Manager (sandbox asíncrono), **no** como snippet en `<head>`.

**Matiz importante:** la premisa del prompt ("gtag.js cargándose en posición demasiado baja del head") no aplica tal cual: GA4 no se carga vía snippet de head sino vía Web Pixel Manager de Shopify (cargado de forma asíncrona y sandboxeada por diseño). No es reposicionable editando `theme.liquid`.

### Acción tomada
**NINGUNA edición de código.** Fuera de scope de Claude Code (decisión humana).

### Instrucción para Bosco
El snippet de GA4 (G-EEFRE4XFMF) lo gestiona la integración nativa Shopify→Google (Web Pixel Manager). No se reposiciona editando el tema. Opciones si se quiere control de posición/carga:
1. **(Recomendado) Dejarlo como está.** El Web Pixel Manager de Shopify ya carga GA4 de forma asíncrona y resiliente; reposicionarlo manualmente puede duplicar eventos y romper el consent mode nativo. La "cobertura 77,5%" probablemente se deba a consent mode / adblockers, no a la posición del tag.
